    Triggers

    Starting a Flow

    Oct 6, 2021

    What is a Trigger?

    A trigger is a method of starting, a Workflow/Task. There are triggers based on a number of different events, as described below:

    Task Instantiated

    This event will occur when a selected Task is created, every time.

    Task Assigned

    This event occurs when the assignee of a task is changed.

    Task Started

    This event occurs when a selected Task is moved to the In Progress step. If the Task is moved from New directly to Completed, this event will not fire.

    Task Completed

    This event occurs when a selected Task is moved to the Completed status, from either New or In Progress. occurs when all tasks in a workflow instance have been moved to the Completed status.

    This event also occurs for notification tasks, after the notification is sent.

    Custom Events

    A custom event is an event published via the api, with an associated routing key. By creating an Event Definition, you can tell your task to be triggered by a custom event.

    This can be used as a way of directing the output of AireForms/forms4health to a Workflow. The AireFlow plugin is already installed in forms4health, so collecting the output of a form can be accomplished simply by using the correct routing key. A stream will then trigger the associated workflow with the form as input data. When a forms4health form is submitted, this is set by convention as {FORM_KEY}-{VERSION}. For example, if you had version 3 of a form with key about-me, then the routing key you should use is about-me-3.

    Mutations

    Mutations allow you to update an existing task. They run under the same conditions a trigger can be run, and additionally include a JSON Patch document which describes how a task should be updated. Mutations must match the trigger conditions and have the same correlation id. If an event is raised without a matching correlation id, it will be ignored.

    The JSON Patch document is the same format as a HTTP Patch. Each item in the array describes a single operation applied to the original task. An example document to change a task name to “new name!” would look like this:

    [
    {
        "op": "add", // "add", "remove", "replace"
        "path": "/name", // "/name", "/state/{key}", "/state", "/assigneeId", "/activityContext/status"
        "value": "new name!" // The new value of the property. Can be any arbitrary JSON, not just a string, as long as it matches the type of the property being patched
    }
]

    The entire JSON document can be templated using liquid syntax, so e.g. the value of the property being updated can be dynamic depending on the event context.
